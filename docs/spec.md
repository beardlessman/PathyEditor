# Техническое задание (ТЗ)

## Разработка клиентского веб-приложения для визуализации GPX-треков

### 1. Общие сведения

**Стек технологий:** React 18+, MobX 6+ (State Management), TypeScript, Leaflet и React-Leaflet, Tailwind CSS (для стилизации).

**Тип приложения:** Client-side SPA (без бэкенда, без авторизации, без персистентного хранения в LocalStorage/IndexedDB).

**Цель:** Создание производительного веб-инструмента для импорта, интерактивного просмотра, анализа и последующего экспорта спутниковых треков в формате GPX.

---

### 2. Функциональные требования

#### 2.1. Модуль импорта (Загрузка файла)

- **Интерфейс:** Зона Drag-and-Drop и кнопка выбора файла через системный проводник.
- **Валидация:** Проверка расширения `.gpx` и валидация структуры XML на клиенте перед парсингом.
- **Парсинг:** Извлечение из XML-структуры тегов `<trkpt>` (широта, долгота), а также метаданных: времени `<time>` и высоты `<ele>` (при наличии).
- **Обработка ошибок:** Вывод тоста или текстового уведомления при загрузке поврежденного или некорректного файла.

#### 2.2. Картографический модуль (OSM & Leaflet)

- **Картооснова:** Использование бесплатных тайлов OpenStreetMap.
- **Рендеринг маршрута:** Отображение трека в виде полилинии (Polyline).
- **Поведение камеры:** Автоматическое центрирование и масштабирование карты (`fitBounds`) по крайним координатам трека сразу после завершения импорта.
- **Маркеры:** Установка уникальных маркеров для начальной точки (Старт) и конечной (Финиш).
- **Интерактивность:** Подсказки (Popup/Tooltip) над треком при клике или наведении, показывающие локальное время точки и высоту.

#### 2.3. Панель метрик и аналитики

**Статистика трека:** Вычисление на клиенте и вывод следующих метрик:

- Общая длина маршрута (в километрах).
- Общее время прохождения маршрута.
- Перепад высот: максимальная, минимальная высота, а также суммарный набор (Elevation Gain) и сброс (Elevation Loss).

**График профиля высот:** Рендеринг интерактивного линейного графика (высота/дистанция).

**Синхронизация:** При наведении на точку графика на карте должен отображаться временный маркер, показывающий, где именно на местности находится эта точка трека.

#### 2.4. Модуль экспорта (Скачивание файла)

- **Выгрузка:** Генерация новой XML-структуры GPX из текущего состояния данных в памяти приложения и скачивание в виде файла.
- **Именование:** Автоматическое формирование имени файла по маске `route_[YYYY-MM-DD_HHMM].gpx`.

---

### 3. Архитектура и задел под редактирование (MobX)

**Масштабируемость:** Структура данных должна быть спроектирована с учётом будущего внедрения функций редактирования (удаление точек, разделение трека, добавление путевых точек WPT).

**Стейт-менеджмент:**

- **RootStore** — главный класс для связи контекстов (на случай добавления UI-сторов, сторов настроек карты и т.д.).
- **TrackStore** — стор, инкапсулирующий логику трека. Массив точек маршрута должен храниться в виде плоского и мутируемого массива объектов (например, `TrackPoint[]`), чтобы в будущем операции вроде `.splice()` для удаления точек автоматически вызывали перерисовку карты.
- **Вычисляемые свойства** (`@computed` / `computed`): расчёт длины трека и перепада высот должен выполняться автоматически через MobX-деривативы при изменении базового массива точек.

---

### 4. Интерфейс и UX

- **Компоновка:** Двухпанельный интерфейс. Левая боковая панель — для загрузки, вывода метрик, графиков и кнопки экспорта. Правая часть — карта Leaflet на 100% оставшейся ширины и высоты экрана.
- **Адаптивность:** Скрытие боковой панели в «гамбургер-меню» или нижнюю шторку на мобильных устройствах.

## ТЗ: Модуль пост-обработки и сглаживания GPX-треков

### 1. Концепция архитектуры данных в MobX

Для реализации цепочки фильтров и возможности отката изменений, структура данных в `TrackStore` должна быть разделена на «сырые» и «финальные» данные:


| Поле                   | Описание                                                                                                                                                                               |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `rawPoints`            | Неизменяемый массив точек, полученный сразу после парсинга GPX (оригинал).                                                                                                             |
| `filterSettings`       | Объект состояния фильтров (включён/выключен, ползунки параметров).                                                                                                                     |
| `points` (`@computed`) | Автоматически вычисляемый массив точек — результат последовательного применения активных фильтров к `rawPoints`. Карта и модуль экспорта подписываются только на этот computed-массив. |


---

### 2. Функциональные требования к фильтрам

Приложение должно поочерёдно применять фильтры в строгом конвейерном порядке (Pipe):

```
rawPoints ➡️ Фильтр Калмана ➡️ Фильтр шума ➡️ Оптимизация (RDP) ➡️ Сглаживание (Чайкин) ➡️ Фильтр остановок ➡️ points
```

#### 2.1. Фильтр 1: Фильтр Калмана

**Назначение:** Устранение высокочастотного GPS-шума и «облаков» точек при остановках с сохранением инерции движения.

**Параметры управления (UI):**

- Чекбокс активации.
- Слайдер «Погрешность измерения (R)» от 1 до 50 метров (шаг 1).
- Слайдер «Динамика движения (Q)» от 0.1 до 5.0 (шаг 0.1).

**Логика работы:** Для широты и долготы независимо применяется линейный фильтр Калмана с состоянием «позиция + скорость». Параметр R задаёт доверие к измерению GPS, Q — готовность фильтра реагировать на реальные повороты.

**Результат:** Трек сглаживается без сильного срезания углов, особенно на низких скоростях.

#### 2.2. Фильтр 2: Шумоподавление (Скользящее среднее / Moving Average)

**Назначение:** Удаление микро-колебаний и «зубцов» на прямых участках.

**Параметры управления (UI):**

- Чекбокс активации.
- Слайдер «Размер окна» (Window Size) от 3 до 99 точек (только нечётные значения, шаг 2).

**Логика работы:** Координаты каждой точки i заменяются на среднее арифметическое координат соседних точек в пределах заданного окна.

#### 2.3. Фильтр 3: Оптимизация (Алгоритм Рамера — Дугласа — Пекера / RDP)

**Назначение:** Радикальное сокращение избыточных точек на прямых отрезках для оптимизации размера файла.

**Параметры управления (UI):**

- Чекбокс активации.
- Слайдер «Порог чувствительности» (Tolerance) от 0.1 до 10.0 метров (шаг 0.1).

**Логика работы:** Использовать библиотеку `simplify-js`. Алгоритм должен исключать точки, отклонение которых от осевой линии маршрута меньше заданного порога.

#### 2.4. Фильтр 4: Геометрическое сглаживание (Алгоритм Чайкина)

**Назначение:** Скругление острых углов на поворотах, создание «эффекта навигатора» (как на референсе).

**Параметры управления (UI):**

- Чекбокс активации.
- Слайдер «Степень сглаживания» (количество итераций) от 1 до 3 (шаг 1).

**Логика работы:** Алгоритм срезает углы полилинии, заменяя каждую вершину двумя новыми точками на расстоянии 25% и 75% по длине ребра. Этот фильтр увеличивает количество точек.

#### 2.5. Фильтр 5: Фильтр остановок

**Назначение:** Удаление GPS-дрейфа при длительных остановках — «облака» точек вокруг одного места схлопываются в одну координату.

**Предусловие:** Фильтр доступен только для треков, в которых **все** точки содержат метки времени `<time>`. При отсутствии временных меток фильтр не применяется; в UI отображается предупреждение.

**Параметры управления (UI):**

- Чекбокс активации.
- Слайдер «Радиус» от 5 до 100 метров (шаг 5, по умолчанию: 40).
- Слайдер «Время» от 1 до 15 минут (шаг 1 минута, по умолчанию: 1 минута).

**Логика работы:**

1. Трек сканируется последовательно. Для каждой стартовой точки кластера алгоритм набирает все последующие точки, расстояние от которых до старта не превышает заданный **радиус**.
2. Вычисляется время пребывания в кластере — разница между временем первой и последней точки кластера.
3. Если время пребывания **не меньше** порога и в кластере больше одной точки, весь кластер заменяется **одной** точкой-центроидом:
  - координаты — среднее арифметическое широты и долготы точек кластера;
  - высота — среднее арифметическое `<ele>` (если есть у всех точек кластера);
  - время — время первой точки кластера.
4. Если порог не достигнут, точки кластера сохраняются без изменений.

**Результат:** На участках длительных стоянок исчезает «шумовое облако» GPS-точек; длина трека и графики скорости становятся реалистичнее. Короткие остановки и паузы ниже порога не затрагиваются.

---

### 3. Требования к интерфейсу (UI/UX)

- **Панель фильтров:** Размещается в боковой панели. Каждый фильтр оформлен в виде отдельного блока с тумблером (Toggle/Checkbox) и интерактивным слайдером для настройки. Порядок применения фильтров в конвейере можно изменить перетаскиванием блоков (drag-and-drop) за иконку ⠿.
- **Информационный виджет:** В реальном времени (под слайдерами) отображает метаданные текущего состояния линии:
  - «Точек в оригинале: XXX ➡️ После фильтров: YYY (Изменение: -ZZ%)»
  - Динамически обновляемая общая длина трека.
- **Превью на карте:** При изменении положения любого слайдера линия трека на Leaflet-карте должна мгновенно перерисовываться (без перезагрузки страницы) благодаря реактивности MobX.
- **Экспорт:** При нажатии кнопки «Экспорт GPX» XML-файл генерируется на основе текущего отфильтрованного массива `points`.

---

### 4. Требования к производительности

- Применение фильтров к треку в 5000+ точек при сдвиге слайдера не должно вызывать фризов интерфейса (микро-лагов UI).
- Если расчёты на клиенте занимают более 100 мс, выполнение конвейера фильтрации должно быть оптимизировано с помощью **debounce** (задержка выполнения расчётов на 150–200 мс после того, как пользователь отпустил ползунок слайдера).

## ТЗ: Авто-заполнение разрывов и объединение сегментов

### 1. Назначение фичи

Инструмент автоматически находит в сессии пользователя изолированные фрагменты треков (сегменты), которые с высокой долей вероятности принадлежат одному непрерывному маршруту, но были разделены из-за временной потери GPS-сигнала. Фича восстанавливает геометрию дорог между ними и объединяет фрагменты в один связный трек.

Фича размещается в боковой панели в отдельном блоке **«Объединение сегментов»** и не входит в синхронный конвейер точечных фильтров.

### 2. Входные данные и параметры

**Вход:** массив загруженных треков (сегментов), где каждый элемент содержит:

- массив точек с координатами `[lat, lon]`;
- временные метки `<time>`;
- дополнительные данные (`<ele>`, скорость — при наличии).

**Параметры (UI):**

| Параметр | По умолчанию | Диапазон |
| -------- | ------------ | -------- |
| Мин. время разрыва | 1 мин | 1–15 мин |
| Макс. время разрыва | 5 мин | 1–15 мин |
| Профиль маршрута | Пешком | Пешком / Велосипед / Авто |

Минимальное время — порог, ниже которого разрыв считается случайным сбоем и **не** объединяется. Максимальное время — критический порог, после которого разрывы трактуются как разные сессии записи.

### 3. Критерии поиска разрывов между сегментами

Алгоритм последовательно перебирает пары соседних треков в хронологическом порядке (конец трека N и начало трека N+1) и проверяет условия:

- **Хронологический порядок:** `Timestamp_{start}(N+1) > Timestamp_{end}(N)`
- **Временной интервал:** `Мин. время ≤ ΔT ≤ Макс. время`, где ΔT — разница между концом первого и началом второго сегмента
- **Разумное расстояние:** `S ≤ V_{max} × ΔT`, где S — расстояние между точками разрыва, V_{max} — максимальная скорость для выбранного профиля

### 4. Предварительная сортировка и фильтрация

Перед кластеризацией все треки проходят этап валидации и сортировки:

- **Временная валидация:** треки без меток `<time>` на всех точках исключаются из анализа (остаются в сессии без изменений).
- **Сортировка:** валидные треки сортируются по возрастанию времени начала:  
  `Timestamp_{start}(Track_1) ≤ Timestamp_{start}(Track_2) ≤ … ≤ Timestamp_{start}(Track_M)`
- **Сложность:** работа только с отсортированным списком даёт линейный поиск кандидатов O(N), так как потенциальные пары находятся рядом.

### 5. Алгоритм выделения групп (кластеризация)

«Жадное» объединение треков в группы на основе скользящего окна:

1. **Инициализация:** `Groups = []`, `CurrentGroup = [Track_0]`.
2. **Проход:** для каждого `Track_i` (i > 0) сравнивается конец последнего трека в `CurrentGroup` с началом `Track_i` по критериям из п. 3.
   - Если условия выполнены — `Track_i` добавляется в `CurrentGroup`.
   - Если нет — при `|CurrentGroup| > 1` группа сохраняется в `Groups`, затем `CurrentGroup = [Track_i]`.
3. **Завершение:** если в последней `CurrentGroup` более одного трека, она также добавляется в `Groups`.

### 6. Пакетное объединение (Pipeline)

```
[N исходных треков]
       │
       ▼ (Сортировка по времени)
[Отсортированный массив]
       │
       ▼ (Кластеризация)
[Массив групп]
       │
       ├─► Группа 1 (3 фрагмента) ──► Роутинг разрывов ──► Слияние в Трек A
       ├─► Группа 2 (2 фрагмента) ──► Роутинг разрывов ──► Слияние в Трек B
       └─► Группа 3 (4 фрагмента) ──► Роутинг разрывов ──► Слияние в Трек C
```

- Для каждой группы применяется роутинг и последовательное слияние: сначала между 1-м и 2-м треком, затем 2-м и 3-м и т.д.
- Запросы к API роутинга выполняются пачками (`Promise.all`) с ограничением параллелизма, чтобы не блокировать UI и не получить ошибку 429 (Rate Limiting).

### 7. Роутинг и интерполяция геометрии

Вместо прямой линии между сегментами используется сервис построения маршрутов по дорожной сети (**Mapbox Directions API**).

- **Запрос:** координаты последней точки трека N (старт) и первой точки трека N+1 (финиш).
- **Профиль:** `walking` / `cycling` / `driving` в зависимости от выбора пользователя.
- **Интерполяция времени:** для каждой точки маршрута  
  `t_i = t_{end}(N) + (t_{start}(N+1) - t_{end}(N)) × (d_i / D_{total})`  
  Высота интерполируется линейно по расстоянию (если есть у обеих крайних точек).

### 8. Объединение данных (MobX & State)

После успешного получения геометрии разрыва:

- Склеивается массив точек: `[Точки трека N] + [Точки роутинга] + [Точки трека N+1]`.
- В `TrackStore` исходные сегменты группы заменяются одним объединённым треком; карта Leaflet перерисовывается реактивно.
- Перед применением сохраняется **снимок** исходных треков; отключение фичи восстанавливает состояние из снимка.

### 9. Требования к интерфейсу (UI/UX)

- **Чекбокс активации** «Авто-заполнение разрывов» в блоке «Объединение сегментов».
- **Слайдеры** мин./макс. времени разрыва и **Select** профиля передвижения.
- **Двухфазный сценарий:**
  1. При включении — поиск групп и **превью** без изменения стора.
  2. Кнопка **«Применить объединение»** — запуск роутинга и фиксация результата в MobX.
- **Превью на карте:** треки в найденных группах перекрашиваются в цвет группы; между концом и началом соседних сегментов отображаются **пунктирные** линии будущих соединений.
- **Прогресс:** прогресс-бар и текстовая индикация этапов, например: «Поиск групп…», «Найдено 3 группы», «Восстановление геометрии: 1 из 3…».
- **Лоадер** на время сетевых запросов к API роутинга (в панели и на кнопках экспорта).
- **Предупреждения:** если у части треков нет меток времени — сообщение о количестве исключённых из анализа файлов.

### 10. Предусловия и ограничения

- Для работы фичи в сессии должно быть **минимум 2 трека** с полными временными метками.
- Токен **Mapbox** (`VITE_MAPBOX_TOKEN`) обязателен для построения маршрутов.
- Треки без `<time>` не участвуют в кластеризации, но сохраняются в итоговом списке после объединения остальных.

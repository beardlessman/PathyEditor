# Техническое задание (ТЗ): Фича «Авто-заполнение разрывов»

## 1. Назначение фичи

Инструмент должен автоматически находить в сессии пользователя изолированные фрагменты треков (сегменты), которые с высокой долей вероятности принадлежат одному непрерывному маршруту, но были разделены из-за временной потери GPS-сигнала. Фича должна восстановить геометрию дорог между ними и объединить фрагменты в один связный трек.

## 2. Входные данные

- **Массив треков (сегментов)**, где каждый элемент содержит:
    - массив точек с координатами `[lat, lng]`
    - временную метку `timestamp`
    - дополнительные данные (скорость, высота)

**Параметры фильтра (задаются пользователем в интерфейсе):**
- **Мин. время разрыва** (по умолчанию: 1 мин) — порог, ниже которого разрыв считается случайным сбоем.
- **Макс. время разрыва** (по умолчанию: 5 мин) — критический порог, после которого разрывы считаются намеренной остановкой записи (разные сессии).
- **Профиль маршрута** (Пешком, Велосипед, Авто) — определяет логику построения геометрии между точками.

## 3. Критерии поиска разрывов

Алгоритм должен последовательно перебирать пары сегментов (конец трека N и начало трека N+1) и проверять условия:

- **Хронологический порядок**:  
  `Timestamp_{start}(N+1) > Timestamp_{end}(N)`

- **Временной интервал**:  
  Разница во времени между концом первого и началом второго сегмента находится в диапазоне:  
  `Мин. время ≤ ΔT ≤ Макс. время`

- **Разумное расстояние**:  
  Расстояние между точками разрыва не превышает максимально возможное расстояние для выбранного профиля за время ΔT:  
  `S ≤ V_{max} × ΔT`

## 4. Алгоритм заполнения геометрии (роутинг)

Вместо соединения точек прямой линией, необходимо использовать сервис построения маршрутов по дорожной сети.

- **Интеграция:** Использовать OSRM API (или альтернативы: GraphHopper, Valhalla).
- **Запрос:** Отправлять координаты последней точки трека N (как старт) и первой точки трека N+1 (как финиш).
- **Профиль:** Передавать соответствующий `profile` (walking, bike, car) в API роутера.
- **Результат:** Полученную геометрию (массив точек) интерполировать по времени между `Timestamp_{end}(N)` и `Timestamp_{start}(N+1)`.

## 5. Объединение данных (MobX & State)

После успешного получения геометрии разрыва:

- Смержить массив точек: `[Точки Трека N] + [Точки Роутинга] + [Точки Трека N+1]`
- Обновить состояние в MobX store, заменив старые раздельные сегменты на один новый объединённый трек.
- Триггернуть перерисовку слоя на карте Leaflet.

## 6. Требования к интерфейсу (UI/UX)

- **Чекбокс активации:** Возможность включить/выключить фильтр в боковой панели (как на скриншоте).
- **Инпуты параметров:** Поля для ввода минут (минимальное и максимальное время) и выпадающий список (Select) для выбора профиля передвижения.
- **Лоадер:** Отображать индикатор загрузки на время выполнения сетевых запросов к API роутинга.

# Дополнение к ТЗ: Группировка и пакетная обработка треков

## 1. Предварительная сортировка и фильтрация

Для эффективной работы алгоритма все загруженные треки должны пройти этап валидации и сортировки:

- **Временная валидация:**  
  Исключить из анализа треки, у которых отсутствуют или повреждены временные метки (`timestamp`).

- **Сортировка:**  
  Отсортировать массив треков по возрастанию времени их начала:  
  `Timestamp_{start}(Track_1) ≤ Timestamp_{start}(Track_2) ≤ ... ≤ Timestamp_{start}(Track_M)`

- **Пул кандидатов:**  
  Работа алгоритма только с отсортированным списком позволяет сократить сложность поиска до линейной (`O(N)`), так как потенциальные кандидаты для слияния находятся рядом друг с другом.

## 2. Алгоритм выделения групп (кластеризация)

Реализация «жадного» объединения треков в группы на основе скользящего окна:

1. **Инициализация:**  
   - Создать новый массив групп: `Groups = []`.
   - Взять первый трек из отсортированного списка и создать с ним начальную группу: `CurrentGroup = [Track_0]`.

2. **Проход по трекам:**  
   Для каждого следующего трека `Track_i` (`i > 0`):

   - Взять последний трек из текущей группы (`LastTrackInGroup = CurrentGroup[CurrentGroup.length - 1]`).
   - Проверить условия связности между концом `LastTrackInGroup` и началом `Track_i`:
     - **Временное окно:**  
       `Мин.время ≤ (Timestamp_{start}(Track_i) - Timestamp_{end}(LastTrackInGroup)) ≤ Макс.время`
     - **Пространственное окно:**  
       Расстояние между финальной точкой `LastTrackInGroup` и стартовой точкой `Track_i` не превышает допустимого значения (для выбранного профиля).
   - **Если условия выполняются:**  
     - Добавить `Track_i` в `CurrentGroup` и продолжить итерацию.
   - **Если условия не выполняются:**
     - Если в `CurrentGroup` более одного трека, сохранить её: `Groups.push(CurrentGroup)`.
     - Создать новую группу: `CurrentGroup = [Track_i]`.

3. **Завершение:**  
   После прохода всех треков, если в последней `CurrentGroup` более одного трека, добавить её в массив групп.

## 3. Порядок выполнения пакетного объединения (Pipeline)

После формирования массива групп (`Groups`, где каждая группа содержит от 2 до X треков) выполняется следующий pipeline:

```
[25+ исходных треков] 
       │
       ▼ (Сортировка по времени)
[Отсортированный массив]
       │
       ▼ (Кластеризация)
[Массив групп (например, 3 группы)]
       │
       ├─► Группа 1 (3 фрагмента) ──► [Роутинг разрывов] ──► Слияние в Трек A
       ├─► Группа 2 (2 фрагмента) ──► [Роутинг разрывов] ──► Слияние в Трек B
       └─► Группа 3 (4 фрагмента) ──► [Роутинг разрывов] ──► Слияние в Трек C
```

- Для каждой группы применяется базовый алгоритм роутинга и слияния.
- Разрывы внутри одной группы обрабатываются последовательно (сначала между 1 и 2-м треком, затем 2 и 3-м и т.д.).
- Запросы к роутингу (OSRM API) выполняются параллельно с помощью `Promise.all` (если это допускается лимитами API), либо через управляемую очередь (`queue`), чтобы избежать блокировки UI и превышения лимита (ошибки 429).

## 4. UI/UX: Индикация и превью

- **Прогресс:**  
  Для обработки 25+ треков требуется много сетевых запросов, поэтому должен отображаться прогресс-бар или шаговая индикация (например, «Поиск групп... Найдено 3 группы. Восстановление геометрии: обработано 1 из 3...»).

- **Превью изменений (опционально):**  
  Перед сохранением изменений в MobX-сторе, визуально выделить на карте Leaflet найденные группы (например, окрасить в один цвет, показать пунктиром предполагаемые соединения), и добавить кнопку «Применить объединение».

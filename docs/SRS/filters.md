# Модуль пост-обработки и сглаживания GPX-треков

## 1. Архитектура данных в MobX

Для реализации цепочки фильтров с возможностью отката изменений структура данных в `TrackStore` должна быть разделена на "сырые" и "финальные" данные:

- **rawPoints** — неизменяемый массив точек, полученный после парсинга GPX (оригинал).
- **filterSettings** — объект состояния фильтров (флаги активности, значения параметров/ползунков).
- **points** (`@computed`) — вычисляемый массив точек, результат последовательного применения активных фильтров к массиву `rawPoints`. Только этот computed-массив подписан на карту и модуль экспорта.

---

## 2. Функциональные требования к фильтрам

Приложение должно поочерёдно применять **три** независимых фильтра в строгом порядке ("конвейер"):

```
rawPoints
  ➡️ Фильтр шума
  ➡️ Оптимизация (RDP)
  ➡️ Сглаживание (Чайкин)
➡️ points (final)
```

### 2.1. Фильтр 1: Шумоподавление (Скользящее среднее / Moving Average)

- **Назначение:** Удаление микро-колебаний и "зубцов" на прямых участках.
- **Параметры в UI:**
  - Чекбокс активации.
  - Слайдер "Размер окна" (от 3 до 15, только нечётные, шаг 2).
- **Логика:** Каждая точка \(i\) заменяется на усреднённые координаты соседних точек в установленном окне.

### 2.2. Фильтр 2: Оптимизация (Рамер — Дуглас — Пекер / RDP)

- **Назначение:** Сильное сокращение избыточных точек на прямых для оптимизации размера файла.
- **Параметры в UI:**
  - Чекбокс активации.
  - Слайдер "Порог чувствительности" (от 0.1 до 5.0 метров, шаг 0.1).
- **Логика:** Используется библиотека `simplify-js`. Алгоритм исключает точки, отклонение которых от линии маршрута меньше установленного порога.

### 2.3. Фильтр 3: Геометрическое сглаживание (Чайкин)

- **Назначение:** Скругление острых углов на поворотах для "эффекта навигатора".
- **Параметры в UI:**
  - Чекбокс активации.
  - Слайдер "Степень сглаживания (итерации)" (от 1 до 3, шаг 1).
- **Логика:** Каждый угол заменяется двумя новыми точками на расстоянии 25% и 75% длины рёбер. Этот фильтр увеличивает количество точек, поэтому всегда последний в цепочке.

## 3. Требования к интерфейсу (UI/UX)

- **Панель фильтров:** Размещается в боковой панели. Каждый фильтр оформлен отдельным блоком с тумблером и слайдером.
- **Информационный виджет:** Под слайдерами в реальном времени отображает:  
    `Точек в оригинале: XXX ➡️ После фильтров: YYY (Изменение: -ZZ%)`  
  + динамически обновляемая общая длина трека.
- **Превью на карте:** При изменении любого слайдера линия трека на карте мгновенно перерисовывается (без перезагрузки страницы, благодаря реактивности MobX).
- **Экспорт:** Кнопка "Экспорт GPX" генерирует XML-файл на основе текущего отфильтрованного массива `points`.

---

## 4. Требования к производительности

- Фильтрация треков в 5000+ точек при изменении слайдера не должна вызывать заметных лагов UI.
- Если расчёты занимают более 100 мс, конвейер фильтрации должен быть оптимизирован с помощью **debounce** (отложенное выполнение расчётов на 150–200 мс после отпускания слайдера).
